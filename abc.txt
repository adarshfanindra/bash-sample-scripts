## 2. Incident Timeline

**Phase 1**

* On Friday around 7 PM IST, the IIQ Task server became unreachable via SSH (connection timeout).
* AWS status checks showed 2/3 checks passing; Instance Status Check failed.
* Serial console access revealed the server had dropped into a dracut emergency shell with:

```
Failed to switch root: Specified switch root path '/sysroot' does not seem
to be an OS tree. os-release file is missing.
```

* The root XFS filesystem (nvme0n1p4) had mounted read-only due to corruption, preventing OS boot.

---

## 3. Root Cause Analysis

**Primary Cause — XFS Filesystem Corruption**

* The root partition (nvme0n1p4, 50GB gp3 EBS volume) suffered XFS filesystem corruption.
* Likely triggers:

  * Abrupt shutdown or kernel panic during active disk writes
  * Disk I/O error during a system operation

**Effect of xfs_repair**

* Running `xfs_repair` moved disconnected inodes (OS directories) to `/var`.
* Critical directories `/etc`, `/usr`, `/bin`, `/sbin`, `/home`, `/lib64`, `/libexec` were physically intact but misplaced.

| Directory   | Status After xfs_repair        |
| ----------- | ------------------------------ |
| /etc        | Moved to /var/etc — recovered  |
| /usr        | Moved to /var/usr — recovered  |
| /home       | Moved to /var/home — recovered |
| /root       | Moved to /var/root — recovered |
| /bin        | Symlink recreated → usr/bin    |
| /sbin       | Symlink recreated → usr/sbin   |
| /lib64      | Symlink recreated → usr/lib64  |
| /opt/tomcat | Intact — no data loss          |
| /var/iga    | Intact — no data loss          |

## 4. Recovery Procedure

**Step 1 — Access via EC2 Serial Console**

* SSH unavailable; accessed dracut emergency shell through AWS EC2 Serial Console.

**Step 2 — Identify Disk Layout**

* Commands: `lsblk`, `blkid`
* Partitions:

  * nvme0n1p2 — /boot/efi (FAT32, ESP)
  * nvme0n1p3 — /boot (XFS)
  * nvme0n1p4 — / root filesystem (XFS)

**Step 3 — Run xfs_repair**

```
xfs_repair /dev/nvme0n1p4
```

* Disconnected inodes moved to `/var`.

**Step 4 — Mount Root Filesystem**

```
mount /dev/nvme0n1p4 /sysroot
```

**Step 5 — Locate Recovered OS Directories**

```
find /sysroot -maxdepth 2 -type d
```

* All OS files confirmed in `/sysroot/var`.

**Step 6 — Restore Missing Directories**

```
mkdir /sysroot/etc && cp -a /sysroot/var/etc/. /sysroot/etc/
mkdir /sysroot/usr && cp -a /sysroot/var/usr/. /sysroot/usr/
mkdir /sysroot/home && cp -a /sysroot/var/home/. /sysroot/home/
cp -a /sysroot/var/root/. /sysroot/root/
mkdir /sysroot/srv /sysroot/mnt /sysroot/media
```

**Step 7 — Recreate RHEL 9 Symlinks**

```
ln -s usr/bin /sysroot/bin
ln -s usr/sbin /sysroot/sbin
ln -s usr/lib64 /sysroot/lib64
ln -s usr/libexec /sysroot/libexec
```

**Step 8 — Mount Boot Partitions and Virtual Filesystems**

```
mount /dev/nvme0n1p3 /sysroot/boot
mount /dev/nvme0n1p2 /sysroot/boot/efi
mount --bind /dev /sysroot/dev
mount --bind /proc /sysroot/proc
mount --bind /sys /sysroot/sys
mount --bind /run /sysroot/run
```

**Step 9 — Chroot into Recovered OS**

```
chroot /sysroot /bin/bash
```

* Verified hostname, disk usage, and OS version.

**Step 10 — Rebuild initramfs and Bootloader**

```
dracut --force
grub2-mkconfig -o /boot/grub2/grub.cfg
```

**Step 11 — Force Reboot**

```
echo b > /proc/sysrq-trigger
```

* Booted successfully.

**Additional Recovery Steps**

* Failed login to corrupted instance; took snapshot and attached to helper instance.
* AWS Support used EC2 Rescue (ec2rs) to recover the volume.
* New instance created from AMI; files copied from recovered volume.
* Tomcat, Java, and SailPoint IIQ restored.

## 6. Recommendations

**Immediate Actions**

1. **Snapshot Policy**

   * Implement automated EBS snapshots via AWS Data Lifecycle Manager (DLM) with minimum 7-day retention.

2. **XFS Filesystem Monitoring**

   * Monitor system logs for XFS errors.
   * Configure CloudWatch alarms for EBS read/write errors and disk utilization above 80%.

---

## 7. Lessons Learned

* XFS filesystem corruption can be recovered without data loss if repair output is carefully analyzed.
* EC2 Serial Console is essential for boot-level recovery; ensure it’s enabled on all production instances.
* Snapshot frequency must align with recovery point objectives; weekly or monthly snapshots are insufficient.
* `xfs_repair` moves disconnected inodes to `/var`, not `lost+found`; check `/var` when directories appear missing after repair.
